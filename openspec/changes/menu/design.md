## Context

`dx` already provides mode-based candidate generation (`dx complete paths|ancestors|frecents|recents|stack`) and selector resolution (`dx navigate up|back|forward [selector]`). Shell hooks are generated by `dx init` and currently rely on shell-native completion UIs and wrapper functions for navigation.

This leaves a UX gap: cross-shell completion behavior is inconsistent, interactive filtering depth varies by shell, and selecting a non-top candidate often requires additional keystrokes or manual selectors. The project direction has already identified an interactive menu path as an opt-in experience that preserves native completion defaults outside dx contexts.

Constraints:
- Preserve current `dx complete` and wrapper contracts; `dx menu` is additive.
- Keep selector/matching logic in Rust rather than shell scripts.
- Ensure clean fallback behavior when no TTY is available or the user cancels.

## Goals / Non-Goals

**Goals:**
- Add a new `dx menu` subcommand that returns a structured edit operation for shell command-line buffers.
- Define a shell-agnostic input/output contract for menu invocation (`buffer`, `cursor`, `cwd`, `session`).
- Reuse existing completion candidate sources and ranking so menu results are consistent with non-menu behavior.
- Support cancellation/no-result flows without mutating shell buffers.
- Add opt-in hook wiring for invoking menu in dx command contexts without breaking default shell completion globally.

**Non-Goals:**
- Replacing `dx complete` or removing existing completion bindings.
- Building a generic fuzzy finder for all shell commands.
- Introducing daemon/background processes.
- Forcing Tab takeover by default across all shells.

## Decisions

### D1: Introduce a dedicated `dx menu` command with structured JSON output

**Choice:** Add `dx menu` as a first-class CLI subcommand that accepts command-line buffer context and prints a JSON action object. Default output format is JSON because shells must apply edits deterministically.

Proposed action schema:

```json
{
  "action": "replace",
  "replaceStart": 3,
  "replaceEnd": 8,
  "value": "/absolute/path"
}
```

Or cancellation/no-op:

```json
{ "action": "noop" }
```

**Rationale:** A structured patch contract isolates shell-specific buffer APIs from selection logic and keeps menu behavior portable.

**Alternatives considered:**
- Plain path-only stdout: insufficient for in-place replacement when cursor is mid-buffer.
- Shell-specific output modes: increases complexity and drift risk.

### D2: Candidate sourcing reuses existing completion modes

**Choice:** `dx menu` maps command context to existing completion modes (`paths`, `ancestors`, `frecents`, `recents`, `stack`) and reuses the same filtering/ranking pipeline.

**Rationale:** Avoids duplicate matching logic and guarantees that menu and non-menu suggestions stay aligned.

**Alternatives considered:**
- Separate menu ranking engine: higher maintenance and likely behavioral drift.

### D3: Menu invocation remains opt-in in hooks

**Choice:** Hook generation adds menu helper functions and optional keybinding registration, but default behavior keeps current completion bindings unless explicitly enabled (for example via init flag or environment toggle).

**Rationale:** Minimizes disruption and avoids breaking established shell completion setups.

**Alternatives considered:**
- Always-on menu keybinding: too invasive and high compatibility risk.
- No hook integration: feature becomes undiscoverable and inconsistent.

### D4: TTY-aware and dependency-aware fallback behavior

**Choice:** If terminal interaction is unavailable or menu rendering fails, hooks fall back to existing `dx complete` behavior (or native completion path) without changing command semantics.

**Rationale:** Maintains reliability in CI, remote sessions, and minimal terminals.

**Alternatives considered:**
- Hard failure on no TTY: degrades usability in common non-interactive contexts.

### D5: MSRV-compatible TUI stack

**Choice:** Use TUI crates pinned to MSRV-compatible versions and avoid dependency upgrades that raise minimum Rust requirements without an explicit MSRV decision.

**Rationale:** Menu integration is user-facing, but build portability remains a project constraint.

**Alternatives considered:**
- Latest TUI crates: may require newer rustc and break build policy.

### D6: Split I/O contract for completion-context interactivity

**Choice:** In shell completion contexts, `dx menu` SHALL keep stdout machine-readable (JSON action payload) while using TTY streams for interactive input and rendering. Shell hooks provide TTY stdin explicitly (for example `</dev/tty`) when completion stdin is non-interactive.

**Rationale:** Completion frameworks commonly capture stdout/stderr, so coupling UI interactivity to captured streams causes immediate `noop` behavior and menu non-display.

**Alternatives considered:**
- Render UI to captured stdout/stderr: breaks interactivity and corrupts completion output parsing.
- Skip menu whenever stdout is captured: defeats menu usage in the primary Tab-completion path.

### D7: Terminal lifecycle safety on every exit path

**Choice:** Interactive menu runtime MUST restore terminal state (raw mode, alternate screen, mouse capture) on all exits: normal selection, Esc cancel, Ctrl+C cancel, draw/read errors, and unexpected runtime failures.

**Rationale:** Partial cleanup leaves shells in a corrupted state with escaped control sequences and broken input handling.

**Alternatives considered:**
- Best-effort cleanup only on happy path: unacceptable UX and high support burden.

### D8: Debuggability without affecting default UX

**Choice:** Provide opt-in diagnostics (environment-gated) for menu invocation path, TTY detection, and action decisions. Default behavior remains quiet for end users.

**Rationale:** Completion-context failures are hard to diagnose without structured runtime signals.

**Alternatives considered:**
- Always-on stderr logs: noisy and can interfere with shell completion behavior.

## Risks / Trade-offs

- [Risk] Terminal UI behavior differs across emulators -> Mitigation: Keep menu rendering minimal, add snapshot/contract tests for output actions, and provide fallback path.
- [Risk] Shell keybinding conflicts with existing user setup -> Mitigation: Keep menu binding opt-in and scoped to dx contexts only.
- [Risk] Added complexity in hook scripts -> Mitigation: Centralize selector logic in Rust; keep hook-side logic to context capture + action application.
- [Trade-off] Additional dependencies increase binary size -> Acceptable for interactive UX gain; bounded by pinned MSRV-compatible versions.
- [Trade-off] Two completion pathways (`dx complete` and `dx menu`) increase conceptual surface area -> Managed by clear dispatch rules and shared candidate engine.

## Migration Plan

1. Implement `dx menu` CLI entry point and buffer/action schema with noop path.
2. Integrate candidate mapping to existing completion modes and selector pipeline.
3. Add opt-in hook glue for Bash/Zsh/Fish/PowerShell to call `dx menu` and apply returned edits.
4. Implement split I/O behavior for completion contexts (TTY input/rendering + JSON stdout contract).
5. Add terminal lifecycle guards to guarantee cleanup on cancel/error paths.
6. Add tests for action schema, cancellation behavior, shell invocation contracts, completion-context interactivity, and terminal-state restoration regressions.
7. Roll out with menu disabled by default; document enablement, fallback behavior, and troubleshooting hooks.

## Open Questions

- Which enablement mechanism should be canonical for first release: init-time flag, env var toggle, or both?
- Should `dx menu` support explicit `--mode` for direct invocation/testing in addition to buffer parsing?
- What minimal fuzzy behavior is needed beyond existing ranking tiers for a good interactive UX?
- Should PowerShell default to native PSReadLine menu while still exposing `dx menu` for parity/testing?
